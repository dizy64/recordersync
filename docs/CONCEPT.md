# 제품 컨셉

## 한 문장 정의

RecorderSync는 카메라가 짧게 나누어 촬영한 영상들과 보이스레코더가 길게 녹음한
고음질 오디오를 소리 자체로 비교해, 신뢰할 수 있는 영상만 외부 음원으로 교체하는
로컬 우선 CLI 도구다.

## 해결하려는 문제

공연·행사·장시간 촬영에서는 다음과 같은 입력이 흔하다.

- 카메라는 녹화 시작과 중지를 반복해 여러 영상 클립을 만든다.
- 보이스레코더는 계속 녹음하지만 파일 시스템 제한으로 약 2GB마다 파일을 나눈다.
- 카메라와 레코더의 시계가 정확히 맞지 않고 장시간에는 clock drift도 생긴다.
- 파일 시각과 길이만으로는 어떤 외부 구간이 어떤 영상과 대응하는지 확정하기 어렵다.
- 잘못된 자동 교체는 원본 카메라음보다 위험하므로 애매한 결과를 억지로 출력하면 안 된다.

RecorderSync는 분할 파일을 하나의 물리 파일로 복사하지 않고 논리 녹음 세션으로
구성한다. 각 영상의 카메라음을 외부 세션의 특징과 비교하고, 최고 후보가 충분히
신뢰할 수 있고 유일할 때만 결과 영상을 만든다.

## 대상 사용자와 대표 흐름

대상은 보이스레코더와 카메라를 동시에 사용하는 1인 제작자 또는 로컬 영상 보관
워크플로다. 기본 흐름은 항상 분석을 먼저 하고 렌더를 나중에 하는 것이다.

1. 한 촬영 단위의 카메라 영상과 레코더 파일을 `VIDEO_DIR`에 둔다.
2. 레코더 파일이 별도 위치라면 `--audio-dir AUDIO_DIR`를 지정한다. 어느 방식이든 여러
   녹음 세션이 함께 있어도 된다.
3. `recordersync analyze`로 세션 구성, 영상별 confidence, 권장 처리 모드를 확인한다.
   중간 단절과 fallback 추천까지 진단하려면 `--partial`을 명시한다.
4. 전체 일치 영상은 기본 `recordersync process`로 출력한다.
5. 부분 일치 영상도 살리려면 `--mode fallback`을 명시한다. 일치 구간은 레코더음,
   나머지는 카메라음으로 출력한다.
6. `ambiguous`, `unmatched`, `error`는 리포트를 보고 입력이나 임계값을 점검한다.
7. 결과 클립은 이후 TubeArchive에서 병합하고 YouTube에 업로드할 수 있다.

## 핵심 용어

| 용어 | 의미 |
|---|---|
| 오디오 조각 | 레코더가 크기 제한 때문에 나눈 단일 파일 |
| 녹음 세션 | 시간상 연속되고 스트림 규격이 같은 오디오 조각 묶음 |
| 영상 클립 | 카메라 내장 오디오를 가진 개별 영상 파일 |
| 특징 타임라인 | 비교를 위해 8kHz mono PCM에서 만든 6대역 log-energy 배열 |
| 매칭 | 영상 특징과 세션 특징의 normalized cross-correlation 결과 |
| peak margin | 최고 후보와 차순위 후보의 상관도 차이 |
| clock drift | 카메라와 레코더 샘플 클럭 차이로 시간에 따라 누적되는 싱크 오차 |

## 확정된 제품 정책

- 기본값은 외부 오디오 완전 교체(`replace`)다.
- 카메라음을 남기고 싶을 때만 사용자가 `mix`와 비율을 명시한다.
- 부분 일치 구간만 레코더음으로 바꾸려면 사용자가 `fallback`을 명시한다. 폴백 구간의
  카메라음 기본 볼륨은 1.0이며 경계는 짧은 crossfade로 연결한다.
- 원본 영상과 외부 레코더 오디오 볼륨은 각각 0.0~1.0 옵션으로 조정한다.
- 애매한 매칭을 자동으로 mix 처리하지 않는다. 출력하지 않고 리포트한다.
- 최소 길이에 못 미치거나 후보가 애매한 부분 구간은 승인하지 않고 카메라음으로 남긴다.
- 여러 카메라가 같은 순간을 촬영할 수 있으므로 서로 다른 영상이 같은 외부 구간에
  매칭되는 것을 허용한다.
- 출력은 회전 메타데이터가 적용된 원본 표시 해상도와 가로·세로 방향을 유지한다.
  고정 캔버스, 박스 레터, 블러 배경을 자동으로 추가하지 않는다.
- 원본 프레임 타임스탬프와 VFR을 유지하며 고정 frame rate로 변환하지 않는다.
- 기본 결과 이름은 원본 stem을 유지하며, 접두사·접미사는 사용자가 요청할 때만 붙인다.
- 리포트 키와 상태값은 자동화 호환성을 유지하고, 사람이 읽는 사유는 한국어를
  기본으로 하되 영어도 선택할 수 있다.
- `analyze`는 사람용 핵심 목록을 기본으로 하고 상세 JSON은 `--json`으로 명시한다.
- 분석 추천은 안내만 제공하며 mix, fallback, overwrite 또는 렌더를 자동 실행하지 않는다.
- 전체 일치는 `replace`, 충분히 길고 넓은 부분 일치는 `fallback`을 추천하고, 짧거나
  분산된 부분 일치와 승인되지 않은 결과는 처리를 보류한다.
- 원본 파일은 수정·이동·삭제하지 않는다.
- 출력 경로가 원본과 같으면 overwrite 요청이 있어도 거부한다.
- 결과가 이미 있으면 `--overwrite` 없이는 덮어쓰지 않는다.
- 분석 리포트와 실제 결과가 자동화에서 구분되도록 종료 코드를 `0/1/2`로 유지한다.
- 비즈니스 정책은 단위 테스트로 FFmpeg·파일 I/O를 격리하고, 실제 도구 경계는 공개 합성
  미디어 E2E로 검증한다.

## 목표와 비목표

### v1 목표

- 한 작업에서 최대 12시간 녹음과 영상 200개 처리
- 여러 2GB 조각과 여러 녹음 세션 자동 구성
- 반복 구간과 낮은 신뢰도에 대해 안전하게 출력 거부
- 장시간 클립의 시작 offset과 clock drift 보정
- 앞뒤 길이 차이와 중간 녹음 단절을 다루는 다중 부분 구간 매칭 및 카메라음 폴백
- TubeArchive와 concat 가능한 표준 MP4 생성
- CLI, JSON 리포트, import 가능한 Python API 제공

### v1 비목표

- GUI/TUI
- 영상 전체 병합 또는 YouTube 업로드
- 원본 파일 아카이브·삭제
- 클라우드 업로드나 서버 처리
- 음성 인식·화자 분리·음원 개선
- 사용자의 실제 미디어를 테스트 fixture로 저장소에 포함
- Windows/Linux 동작 보장

## 성공 기준

- 명확한 동일 음원은 기본 임계값에서 `matched`가 된다.
- opt-in 부분 분석은 승인된 구간을 겹치지 않는 `segments`로 보고하고, fallback 렌더는
  그 구간만 레코더음으로 교체한다.
- 반복되거나 비슷한 후보가 둘 이상이면 `ambiguous` 또는 `unmatched`가 된다.
- 매칭 실패 영상에는 출력 파일이 생기지 않는다.
- 렌더 실패나 중단 시 완성되지 않은 파일이 최종 이름으로 남지 않는다.
- 12시간·200영상 합성 벤치가 총 600초, peak RSS 2GB 이내다.
- 모든 비즈니스 정책은 외부 프로세스 없는 단위 테스트로 재현된다.

## TubeArchive와의 관계

RecorderSync가 책임지는 것은 녹음 세션 구성, 오디오 구간 매칭, 개별 영상 렌더다.
TubeArchive는 결과 클립 정렬·병합·분할·프로젝트 관리·YouTube 업로드를 책임진다.

가장 단순한 연동은 RecorderSync가 표준화한 개별 결과 파일들을 TubeArchive 입력으로
전달하는 방식이다. TubeArchive가 매칭 결과만 직접 소비하려면 전체 매칭의 `session_id`,
시작점, `tempo_ratio`와 부분 매칭의 다중 `segments`를 모두 이해해야 한다. 각 구간의
세션은 여러 오디오 조각을 concat 입력으로 다뤄야 한다. 첫 번째 조각 경로만 기존
`external_audio_path`에 넘겨서는 조각 경계나 중간 단절을 넘는 클립을 올바르게 렌더할
수 없다.
